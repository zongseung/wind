# BARAM 2026 풍력 발전량 예측 수행 검토 및 기획서

검토일: 2026-07-06 KST

## 1. 결론

현재 `/root/wind/open` 데이터만으로 BARAM 2026 풍력 발전량 예측 프로젝트는 수행 가능하다.
다만 `/root/wind/baram2026_design.pdf`의 설계를 그대로 100% 구현하기에는 제공 데이터와 실행 환경에
차이가 있으므로, 일부 설계는 축소 또는 대체해야 한다.

핵심 판단은 다음과 같다.

| 항목 | 판단 | 근거 |
|---|---|---|
| 예측 과제 수행 | 가능 | 2022-2024 학습 라벨, 2025 평가 NWP, 제출 양식이 모두 존재 |
| 데이터 누설 없는 추론 | 가능 | 테스트 구간에는 LDAPS/GFS만 사용하면 규칙 충족 |
| PDF의 HMM + GRU 하이브리드 | 조건부 가능 | 개념은 가능하나 현재 환경에 ML 패키지가 없고 일부 SCADA 변수가 없음 |
| SCADA 기반 파워커브 적합 | 가능 | 10분 단위 터빈별 발전량, 풍속, 풍향은 제공됨 |
| 1초 SCADA 기반 라벨 정제 | 불가 | 현재 공개 데이터에는 yaw, pitch, RPM, 1초 운전변수가 없음 |
| 제출 파일 생성 | 가능 | `sample_submission.csv` 8,760행과 테스트 NWP 기간이 일치 |

따라서 첫 구현 목표는 PDF의 전체 고급 구조를 한 번에 구현하는 것이 아니라, 누설 없는 강한
표형 모델 기준선을 먼저 만들고, 검증 점수가 확인되는 경우 HMM, GRU, FICR 보정을 단계적으로
추가하는 방식이 적절하다.

## 3. PDF 설계 검토

### 3.1 그대로 채택 가능한 부분

- 예측 시점 정렬: `data_available_kst_dtm` 기준으로 전날 13:00 KST에 사용 가능한 다음날 24시간 예보만 쓰는 구조는 현재 데이터와 일치한다.
- 추론 입력 제한: 평가 기간에는 LDAPS/GFS만 사용한다는 원칙이 맞다.
- 물리 특징: u/v 성분을 풍속, 풍향, 밀도 보정 풍속, 전단 지수, 허브 높이 추정 풍속으로 변환하는 접근은 유효하다.
- 그룹별 설비용량 클리핑: 예측값을 0 이상, 설비용량 이하로 제한하는 것은 필요하다.
- 그룹 3 데이터 부족 대응: 단일 공유 모델과 그룹 임베딩을 쓰는 아이디어는 타당하다.
- FICR 후처리: 6%, 8% 오차 경계를 의식한 선택적 보정은 실험 가치가 있다.

### 3.2 수정이 필요한 부분

| PDF 내용 | 현재 데이터와의 차이 | 수정안 |
|---|---|---|
| 1초 SCADA, yaw, pitch, RPM 사용 | 공개 데이터에 없음 | 10분 SCADA의 발전량/풍속/풍향만 활용 |
| SCADA 기온 사용 | SCADA 파일에 기온 없음 | NWP 기온으로 공기밀도 계산 |
| LDAPS/GFS 모두 80m/100m 풍속 사용 | GFS에는 80m/100m, LDAPS에는 10m/50m 계열 중심 | GFS는 80/100m, LDAPS는 10/50m 기반 전단 추정 |
| HMM `hmmlearn` 전제 | 현재 환경에 패키지 없음 | 초기는 LightGBM/CatBoost 기준선, 이후 의존성 설치 후 HMM 실험 |
| Seq2Seq-GRU 전제 | 현재 환경에 PyTorch/TensorFlow 없음 | 표형 모델 성능 확인 후 PyTorch 환경 구성 |
| 라벨 정제에 운전변수 사용 | 운전변수 미제공 | 라벨/SCADA 이상치 규칙 기반 필터링으로 대체 |

### 3.3 현재 실행 환경

`pyproject.toml`에는 의존성이 비어 있고, 현재 Python 환경에서 다음 주요 패키지는 확인되지 않았다.

- `numpy`
- `pandas`
- `scikit-learn`
- `scipy`
- `hmmlearn`
- `torch`
- `lightgbm`
- `xgboost`
- `catboost`
- `polars`
- `pyarrow`

실제 모델 학습을 시작하려면 의존성 설치가 선행되어야 한다.

## 4. 수행 전략

### 4.1 1단계: 재현 가능한 데이터 파이프라인

목표는 모든 원천 파일을 누설 없이 시간 단위 학습 테이블로 변환하는 것이다.

주요 작업:

- `forecast_kst_dtm`, `data_available_kst_dtm`, `grid_id` 기준으로 GFS/LDAPS를 검증한다.
- GFS 9개 격자, LDAPS 16개 격자를 시간별 wide feature로 변환한다.
- u/v 바람 성분에서 풍속, 풍향 sin/cos, 풍속 제곱/세제곱을 생성한다.
- 격자별 원시값과 전체 격자 통계량을 함께 만든다.
- `info.xlsx`의 터빈 위치를 이용해 근접 격자 가중 평균 특징을 추가한다.
- `train_labels.csv`와 조인하여 그룹별 학습 행을 만든다.
- 제출 생성 시 `sample_submission.csv`의 `forecast_id`, `forecast_kst_dtm`은 변경하지 않는다.

산출물:

- `data/processed/train_features.parquet`
- `data/processed/test_features.parquet`
- `data/processed/feature_schema.json`
- `submission_baseline.csv`

### 4.2 2단계: 강한 기준선 모델

PDF의 HMM/GRU보다 먼저 표형 모델 기준선을 만든다. 이 기준선은 구현 속도와 검증 안정성이 높고,
이후 고급 모델의 성능 비교 기준이 된다.

권장 모델:

- LightGBM 또는 CatBoost 회귀
- 타깃은 그룹별 별도 모델 또는 `group_id`를 포함한 공유 모델
- 손실은 MAE 또는 Huber 계열
- 예측 후 `0 <= yhat <= capacity_kwh` 클리핑

주요 특징:

- GFS 80m/100m 풍속, 풍향, 전단
- LDAPS 10m/50m 기반 풍속, 풍향, 전단 대리값
- 기온, 습도, 기압, 운량, 강수, gust
- 격자별 값, 격자 평균/최대/최소/표준편차
- 월, 일, 시각, 계절 sin/cos
- 리드타임 16~39
- 그룹 설비용량, 그룹 임베딩 또는 group id
- SCADA로 추정한 그룹별 파워커브 예측값

### 4.3 3단계: 물리 기반 파워커브

SCADA는 추론 입력으로 쓰지 않고, 학습 구간에서 NWP 풍속을 발전량으로 변환하는 사전 지식 생성에
사용한다.

작업 내용:

- 10분 SCADA를 시간 단위로 집계한다.
- 터빈별 발전량을 그룹별로 합산한다.
- 풍속 구간별 평균 발전량으로 경험적 파워커브를 만든다.
- cut-in, rated, cut-out 조건을 반영해 단조화한다.
- NWP 허브고도 추정 풍속을 파워커브에 넣어 `pc_pred_group_*` 특징을 만든다.

주의:

- SCADA 발전량 컬럼의 원천 단위가 `kW10m`로 표시되어 있으므로, 시간 단위 집계 규칙을 검증해야 한다.
- KPX 라벨과 SCADA 합산 발전량의 스케일을 비교하여 보정 계수를 둔다.

### 4.4 4단계: HMM 국면 모델

기준선이 마련된 뒤 HMM을 추가한다.

목표:

- 풍속, 풍향, 전단, 난류 대리값, 기온, 기압을 이용해 숨은 국면을 추정한다.
- 국면 확률을 표형 모델 또는 GRU의 입력 특징으로 추가한다.

구현 방안:

- `hmmlearn` 또는 `pomegranate` 설치 후 Gaussian HMM 사용
- 그룹별/계절별로 K=3~6을 탐색
- BIC와 시간 검증 점수를 함께 보고 K 선택
- 국면 확률이 검증 점수를 올리지 못하면 제거

### 4.5 5단계: Seq2Seq-GRU 잔차 보정

GRU는 기준선과 HMM이 만든 예측의 잔차를 보정하는 선택적 고급 계층으로 둔다.

조건:

- PyTorch 환경 구축
- 하루 24시간 단위 시퀀스 데이터셋 구성
- 기준선 예측값이 out-of-fold 방식으로 생성되어야 함

입력:

- 24시간 NWP 시퀀스
- 물리 특징
- 기준선 예측값
- HMM 국면 확률
- 그룹 id 또는 그룹 임베딩

출력:

- 그룹별 시간별 정규화 잔차
- 최종 예측 = 기준선 예측 + 설비용량 * 잔차

도입 기준:

- 2024년 검증 또는 계절 검증에서 LightGBM/CatBoost 앙상블보다 총점이 올라갈 때만 유지한다.

### 4.6 6단계: FICR 후처리

정확도 모델이 안정화된 뒤, 6%와 8% 오차 경계에 대한 후처리를 실험한다.

실험안:

- 검증 잔차 분포를 월/풍속/그룹/리드타임별로 추정한다.
- 예측 불확실성이 큰 시간대는 보수적으로 조정한다.
- FICR 대리 목적함수는 검증 점수 증가가 확인될 때만 적용한다.

주의:

- FICR 보정은 1-NMAE를 악화시킬 수 있으므로 전체 적용하지 않는다.
- 6~8% 경계 근처 또는 발전량이 큰 시간대 중심으로 제한한다.

## 5. 검증 설계

랜덤 분할은 금지한다. 시간 순서와 계절성을 보존해야 한다.

권장 검증:

| 검증명 | 학습 | 검증 | 목적 |
|---|---|---|---|
| Holdout 2024 | 2022-2023 | 2024 | 실제 미래 예측 성능 확인 |
| Group 3 전용 | 2023 | 2024 | 그룹 3 데이터 부족 리스크 확인 |
| 계절 블록 CV | 각 연도 일부 월 holdout | 봄/여름/가을/겨울 | Public/Private 계절 편향 대비 |
| 월별 backtest | 과거 전체 | 다음 월 | 월별 취약 구간 분석 |

평가 구현:

- 유효 시간대는 실제 발전량이 설비용량의 10% 이상인 구간만 포함한다.
- NMAE 분모는 MW가 아니라 시간당 설비용량 kWh를 사용한다.
- 예측값은 평가 전에 그룹별 설비용량으로 클리핑한다.
- 전체 점수는 1-NMAE와 FICR를 모두 계산하되, 모델 선택은 우선 NMAE 중심으로 한다.

## 9. 성공 기준

최소 성공 기준:

- `sample_submission.csv`와 동일한 행 수, 동일한 `forecast_id`, 동일한 `forecast_kst_dtm`의 제출 파일 생성
- 모든 예측값이 0 이상, 그룹별 시간당 설비용량 이하
- 테스트 시점에서 `train_labels`, 평가 기간 SCADA, 사후 정보 사용 없음
- 2024 holdout 기준으로 단순 월/시각 평균 baseline보다 낮은 NMAE 달성

권장 성공 기준:

- LightGBM/CatBoost 기준선 대비 SCADA 파워커브 또는 HMM 특징의 검증 성능 개선 확인
- 그룹 3 전용 검증에서 과적합 없이 안정적인 성능 확보
- 최종 제출 후보 3개 이상을 검증 근거와 함께 관리

## 10. 최종 권고

이 프로젝트는 현재 데이터로 수행 가능하지만, PDF 설계는 연구형 고급 설계에 가깝다. 실제 성적을
목표로 한다면 다음 순서가 가장 실용적이다.

1. 누설 없는 표형 기준선과 metric부터 만든다.
2. SCADA는 테스트 입력이 아니라 파워커브와 학습 품질 개선에만 쓴다.
3. HMM은 국면 확률 특징으로 작게 붙여 ablation으로 검증한다.
4. GRU 잔차 보정은 기준선이 안정화된 뒤 선택적으로 시도한다.
5. FICR 최적화는 마지막 후처리 단계에서만 제한적으로 적용한다.

이 순서로 진행하면 PDF의 핵심 아이디어를 살리면서도, 현재 공개 데이터와 실행 환경의 제약 안에서
재현 가능한 제출물을 만들 수 있다.
